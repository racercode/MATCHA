# MATCHA — Match with Agent

> **AI Agent 代理社交，讓資源主動找到你**
>
> 2026 YTP 黑客松 · 賽題 B · 行善台北 | 隊伍：PV=NTR

---

## 0. 開場：一個迷惘的晚上

小雅，中文系大三，對未來完全沒有概念。同學們開始談實習、談研究所，她連自己想做什麼都不知道。

某天晚上，她打開台北市青年局網站，看到 47 個計畫連結，每個名字都很長，每個頁面都很複雜。她點了兩個，看不懂申請條件，也不確定自己符不符合。然後她關掉了瀏覽器，躺回床上，繼續看著天花板迷茫。

**這不是個案。這是無數台北青年的日常。**

政府擁有豐富的資源——實習媒合、轉職補助、創業貸款、職涯諮詢——但青年的搜尋成本太高，資訊不對稱讓這些資源形同虛設。而在另一端，政府第一線的職涯輔導人員每天接應大量青年，但常常專業不對口——一個擅長創業輔導的承辦人，面對想轉職設計的文組青年，很難給出最精準的建議。人力諮詢有上限，但青年的迷惘沒有。

## 1. 問題分析

### 1.1 痛點拆解

| 角色 | 痛點 | 現況 |
|------|------|------|
| 迷惘青年 | 不知道自己想做什麼，不知道該怎麼尋找工作，也不知道政府有哪些資源可以幫助自己 | 面對大量政策計畫，存在嚴重資訊不對稱，搜尋成本極高，完全不知從何下手 |
| 政府人員 | 擁有資源卻無法有效對接迷惘青年，專業不對口、人力不足 | 第一線人員難以判斷優先級，總是被動等待青年上門，且職員不一定能接應到自己專長能幫助到的青年 |

### 1.2 為什麼這個問題很重要？

台北市每年投入大量預算於青年職涯發展，但政策觸及率始終低於預期。**問題的核心不是資源不足，而是資源與需求之間的「媒合失敗」。** 當一個青年因為搜尋成本太高而放棄，不僅是個人損失，更是整個社會的損失。

### 1.3 問題挑戰在哪？

- **如何深入了解使用者**：青年自己常常不確定自己要什麼，傳統問卷難以捕捉真實需求
- **如何做資源媒合**：政府資源分散、資料量大且異質性高（實習、補助、課程、貸款），難以用單一演算法匹配
- **如何讓 App 易用**：青年對官方 App 的耐心極低，介面必須像社群軟體一樣親切
- **如何讓政府端受益**：第一線人員需要減負，工具需自動化，且讓一線人員精準接觸到需要幫忙的個案

## 2. 動機與核心洞察

### 2.1 一個關鍵觀察

想一想你人生中那些最關鍵的機會——是搜尋引擎給你的，還是某個人在某個場合剛好推薦給你的？

大多數人的答案是後者。**資源的流動依賴社群和人際關係，而不是搜尋欄。** 這也是全球最大的職涯社群 LinkedIn 的核心理念：你加入一個社群，認識了幾個人，某天對方掌握了一個機會，剛好想到你，就推薦過來了。這種「被發現」的互動過程，比主動查詢更精準、更省力，也能接觸到更多使用者不知道的資源。

### 2.2 倚靠人際關係網的問題

但問題在於：維護人際關係需要時間和社交能量。對於正在迷惘的青年而言，這是一道很高的門檻。而對政府機關而言，也很難用社交的方式去與每一個青年建立關係並服務。

### 2.3 核心想法

> **想法很簡單：讓 AI Agent 替你去社交。**

我們的解法不是再做一個「更好的搜尋引擎」，而是建立一個**「代理社交」**的系統。每個青年擁有一個專屬的 AI Agent，它代替你去認識人、經營關係、搜集資訊；每個政府資源也各自有自己的 Agent，主動尋找符合條件的青年。這些 Agent 在一個共享的社群空間中互動、媒合、交換資訊，就像一個 24 小時運作的職涯社群。

**從「被動查詢」轉為「主動媒合」——這就是 MATCHA 的核心理念。**

## 3. 產品原型

### 3.1 使用情境：小雅的一天

小雅下載了 MATCHA。第一次開啟時，她的專屬 Agent「媒伴」開始和她聊天：

> 「嗨！我是你的媒伴。先聊聊吧——你現在大概是什麼狀態？」

小雅說：「剛畢業，讀中文系，想跨去做設計，但不知道從哪裡開始。」媒伴不急著推薦任何計畫，而是繼續問：「聽起來你對設計有感覺，但還在探索方向。你說的設計比較偏視覺、產品、還是品牌這種感覺？」

**幾輪對話後，媒伴建立了小雅的初步畫像**：中文系畢業、對品牌設計和排版有興趣、住台北大安區、正在探索階段。

對話結束後，媒伴邀請小雅玩一下**職涯羅盤**：畫面上出現幾張卡片——「你比較想先累積作品集，還是先找實習？」「你能接受的通勤時間？」「薪資還是學習機會，哪個優先？」小雅像滑交友軟體一樣，幾秒鐘就回答了五個問題，媒伴的推薦精準度從 68% 跳到 85%。

**幾小時後，小雅收到一則通知：「3 個資源主動找上你」**——包括一個創意實習媒合平台（符合度 87%）、一個文組轉職培訓補助（符合度 92%）、一個職涯探索心理測驗（符合度 74%）。每筆資源都附上截止日期和一句話摘要。她不需要搜尋任何東西，資源就這樣找到了她。

小雅對「文組轉職培訓補助」很感興趣，直接在通知下方追問：「申請截止日期是什麼時候？需要準備什麼文件？」AI Agent 查閱該計畫的完整文件後即時回答。政府承辦人在 Dashboard 上看到這筆高分媒合，點了「開啟對話」——小雅收到通知，進入真人聊天室，直接跟承辦人一對一溝通。

**在小雅迷茫這麼多資源、自己該怎麼做時**，發現頁面下方有一個「Coffee Chat」的區塊亮起了紅點。原來媒伴已經幫她找到三位背景相似的使用者——一位同樣是文組轉設計、已經在接案的學姐；一位剛完成職訓課程的社會新鮮人。她讀到這些去識別化的聊天摘要，突然發覺原來這條路可以這樣走，並且真的有人走過，焦慮感少了一大半。

**睡前，小雅點開自己的個人檔案**，看到媒伴對她的理解程度：「職涯方向」72%、「技能偏好」58%、「生活條件」45%。頁面提示她：「再聊兩輪，我就能幫你找到更精準的資源。」小雅覺得自己正在被一點一點地理解，明天還想再打開這個 App。

### 3.2 五大介面

| 介面 | 功能 | 設計理念 |
|------|------|---------|
| **聊天介面** | 與專屬 AI Agent「媒伴」對話，漸進式建立使用者畫像 | 以自然對話取代搜尋，降低搜尋門檻 |
| **通知介面** | 接收來自政府資源 Agent 的媒合通知，顯示符合度與理由 | 資源主動找人，而不是人去找資源 |
| **職涯羅盤** | 類似交友軟體的 Swipe 互動，快速確認偏好與方向 | 回答幾個問題就能深入了解使用者畫像，提升推薦精準度 |
| **Coffee Chat** | 查看 Agent 代為進行的「社交」結果，發現相似背景的人怎麼做 | 同儕經驗是政府資源以外的重要支持 |
| **個人檔案** | 查看檔案完整度、AI 對你的理解程度 | 讓使用者看到自己「被理解」的程度，驅動持續互動 |

### 3.3 政府端介面

- **社群頻道**：類似 Discord/Slack 的頻道，多個使用者 Agent 與多個政府資源 Agent 在其中互動，政府人員可監看需求趨勢
- **媒合與介入列表**：資源 Agent 媒合到的使用者自動提交到列表，由職員判斷是否需要人工介入
- **數據反饋儀表板**：彙總青年諮詢趨勢、資源熱度、媒合成功率等數據，供政策制定者參考

## 4. 系統架構與技術方法

### 4.1 架構總覽

MATCHA 的系統架構由四個核心模組組成，形成一個多 Agent 協作的生態系統：

| 模組 | 職責 | 技術實現 |
|------|------|---------|
| **政府資料前處理** | 整理不同來源的政策資料（青年局、勞動部等），清理並結構化 | PDF/HTML/XLSX parser → Firestore `gov_resources/{rid}/documents` |
| **User Agent（媒伴）** | 與使用者互動探索畫像；定期廣播到社群；接收媒合結果並整理給使用者；尋找相似使用者進行 Coffee Chat | Claude Managed Agent + Persona Profiling Tools |
| **政府資源 Agent** | 接收 User Agent 廣播，讀取資源文件判斷是否符合條件，將結果發送回去 | Claude Managed Agent + RAG (query_resource_pdf) |
| **社群互動模組** | 文字交流頻道，讓 Agent 們發送訊息、尋找匹配、交換資訊 | Channel (Firestore channel_messages + channel_replies) |

### 4.2 媒合流程

```
Step 1 — User Profiling
  使用者透過自然對話與職涯羅盤，User Agent 漸進式建立畫像

Step 2 — 社群廣播
  User Agent 向社群頻道發送「現況廣播」，包含去識別化的使用者狀態摘要

Step 3 — 資源媒合
  政府資源 Agent 接收廣播，讀取資源文件比對條件
  → 符合則主動推送媒合結果（score 0–100 + 理由）
  → 記錄到媒合與介入列表

Step 4 — 通知與確認
  User Agent 整理收到的媒合結果，以易懂的方式通知使用者
  使用者可追問細節（Follow-Up Agent 查閱資源文件回答）

Step 5 — Coffee Chat
  User Agent 掃描社群中相似使用者，配對後促成同儕交流

Step 6 — 人工介入
  政府職員查看媒合列表，對需要專業協助的個案接手

Step 7 — 數據反饋
  政策制定者根據數據儀表板分析趨勢，優化資源配置
```

### 4.3 為什麼用分散式 Agent，而不是一個大模型搜全部？

傳統做法是用搜尋引擎或單一 LLM 在大量資料庫裡搜索，但當 context 太大時，會產生 **context rot** 問題——搜索的精度會下降。

MATCHA 的做法不同：**每個 Agent 只負責一小塊 context**。每個政府資源有自己的 Agent，只需精通自己那份資源的申請條件和文件內容；每個青年也有自己的 Agent，只需深入了解這一個人。媒合發生在 Agent 與 Agent 之間的互動，而不是一個巨大模型試圖同時理解所有人和所有資源。

**優勢：**
- **媒合更精準**：每個 Agent 處理小範圍 context，不因單一模型負擔過重而降低品質
- **算力集中在政府端**：主要推論負擔在政府資源 Agent，政府可控制資源分配
- **水平擴展**：新增一個政府資源，只需新增一個 Agent + 上傳文件，不需修改系統

**限制（誠實揭露）：**
- **非即時回應**：Agent 需要時間互動，使用者的請求需要一小段時間才取得結果。這是我們用「通知」而非「即時搜尋」的原因——媒合品質優於速度
- **初期冷啟動**：社群人數較少時，Coffee Chat 功能價值較低。我們以政府資源媒合作為核心價值，社群功能作為增值

### 4.4 AI Agent 技術細節

MATCHA 使用 **Claude Managed Agents**（Anthropic Sessions API）。每個 Agent 是一個有持續記憶的 AI 實體：

| 組件 | 生命週期 | 用途 |
|------|---------|------|
| **Environment** | 一次性建立 | 雲端容器設定 |
| **Agent** | 持久，可版本更新 | 模型 + system prompt + tools 定義 |
| **Session** | 每使用者/資源，Redis TTL 24h | 對話記憶，checkpoint 狀態 |

```
Redis session 快取：
  session:persona:{uid}       → 一位市民一個 Persona Agent session
  session:gov:{resourceId}    → 一個政府資源一個 Gov Agent session
  session:coffee:{threadId}   → 一個 Peer Thread 一個 Coffee Agent session
```

每個 Agent 的 Tool（技能）對應具體的 Firestore 讀寫：

| Agent | Tool | 做什麼 |
|-------|------|--------|
| Persona Agent | `get_my_persona` | 讀取使用者 persona |
| | `update_persona` | 更新 persona 摘要 |
| | `publish_to_channel` | 廣播到 Channel → 觸發 Gov + Coffee Agent |
| Gov Agent | `query_resource_pdf` | 讀取資源 PDF 文件內容 |
| | `assess_fit` | 評估媒合度 0–100 + 理由 |
| | `write_channel_reply` | 寫入媒合結果 |
| Coffee Agent | `read_channel_messages` | 掃描近期市民摘要 |
| | `propose_peer_match` | 建立配對 |
| | `relay_message` | 代理轉發訊息 |

### 4.5 通訊架構

| 情境 | 方式 | 原因 |
|------|------|------|
| Persona Chat、Coffee Chat、Human Thread 的訊息 | **WebSocket** | 需要即時串流 |
| 媒合結果、新配對、新對話 | **HTTP Polling** | 不需即時，降低複雜度 |

### 4.6 資料層

```
Firestore
├── personas/{uid}                         ← 市民的 AI 整理摘要
├── channel_messages/{msgId}               ← Persona Agent 廣播的市民需求
├── channel_replies/{replyId}              ← Gov Agent 的媒合回覆（含 matchScore）
├── human_threads/{tid}                    ← 政府承辦人開啟的真人對話
│   └── messages/{mid}
├── peer_threads/{tid}                     ← Coffee Agent 配對的市民對話
│   └── messages/{mid}
├── gov_resources/{rid}                    ← 政府資源定義
│   └── documents/{docId}                  ← 資源文件文字（PDF 解析後）
├── match_stats/{resourceId}               ← 各資源的媒合統計
└── gov_agent_runs/{messageId}             ← 冪等觸發紀錄
```

## 6. 落地潛力

### 6.1 台北市應用價值

MATCHA 直接回應「行善台北」賽題的核心目標：

- **對青年**：降低資源搜尋成本至接近零，從「自己找」變成「被找到」
- **對政府**：精準媒合提升政策觸及率並減輕第一線人員負擔，大量對話分析獲得即時的青年需求數據反饋
- **對社會**：形成青年互助社群，讓過來人的經驗傳承給後來者

### 6.2 擴展性

MATCHA 的多 Agent 架構具備良好的擴展性。新增一個政府資源只需新增一個 Agent + 上傳文件，不需修改系統。未來可從台北市擴展到全台，甚至接入民間企業（實習機會、培訓課程），生態自然成長。

### 6.3 商業成本與效益

- **算力套利人力**：以低廉的雲端推論成本，大幅減輕第一線人員負擔，有效替代高昂的真人諮詢工時
- **活化政策預算**：透過自動化媒合精準提升政策觸及率，解決資源閒置的痛點，極大化政府預算實質影響力

## 7. 評分維度分析

### 技術實作 / 功能完整度（25%）

| 面向 | 我們做了什麼 |
|------|-----------|
| **AI 應用深度** | 使用 Claude Managed Agents（Sessions API）實作三個獨立 Agent，非單純 prompt wrapper。每個 Agent 有持久記憶（Session + Redis TTL 24h）、自有工具集（Firestore CRUD）、獨立 system prompt。Gov Agent 支援 RAG：讀取政府 PDF 文件，結構化評估媒合度（0–100 + 理由 + 缺漏資訊）。Follow-Up Agent 支援多輪追問，session 複用。 |
| **系統穩定性** | 冪等觸發機制（`gov_agent_runs/{messageId}`）防重複處理；可控並行度（`GOV_AGENT_CONCURRENCY`）；In-memory fallback 模式讓前端可獨立開發。 |
| **GitHub 開發規範** | pnpm monorepo + shared-types contract-first；三組平行開發透過 mock server 解耦；`.env.example` 完備；功能模組化（agents / routes / lib / ws）。 |

### 創新性（25%）

| 面向 | 我們的差異 |
|------|---------|
| **核心創新：代理社交** | 不是「更好的搜尋引擎」——根本性翻轉資源流動方向：從「人找資源」變成「資源找人」。靈感來自 LinkedIn 觀察：人生關鍵機會來自社群推薦，而非主動搜尋。 |
| **分散式 Agent 架構** | 傳統 LLM 搜索 context 太大會 rot；MATCHA 每個 Agent 只精通一小塊 context，媒合發生在 Agent 間互動。新增資源 = 新增 Agent，水平擴展。 |
| **Coffee Chat：同儕連結** | 不只「人↔資源」，還做「人↔人」媒合——讓迷惘青年看到跟自己相似的人怎麼走過來，提供傳統系統無法給予的同儕支持。 |

### 體驗與可用性（20%）

| 面向 | 設計選擇 |
|------|---------|
| **UI 友善** | 市民端以「聊天」為核心互動，零學習成本。職涯羅盤模仿交友軟體 Swipe 設計。媒合結果以卡片 + 百分比 + 一句話理由呈現。 |
| **數據視覺化** | 政府 Dashboard：媒合統計（回覆數、平均分、開話率）、分數分佈、Resource Agent 數量、單一資源統計。 |
| **弱勢包容性** | 設計初衷就是服務「不知道自己要什麼」的迷惘青年——這群人在傳統搜尋模式下完全被遺漏。對話式互動降低資訊素養門檻：不需要知道「正確的關鍵字」就能被找到。 |

### 主題契合度（20%）

| 面向 | 具體呼應 |
|------|---------|
| **行善台北** | 直接針對台北市青年局的真實政策計畫做媒合（青創貸款、共享空間補助、實習津貼、留學貸款） |
| **開放資料整合** | 資料來源：台北市青年局官網公開文件 + data.taipei。政府端可自行上傳 PDF/Markdown，後端自動解析為文字——不依賴靜態爬蟲，支援動態更新 |

### 落地 / 商業化潛力（10%）

| 面向 | 分析 |
|------|------|
| **實用性** | 完整端到端流程：建立 persona → 自動媒合 → 追問細節 → 承辦人開啟對話。資源管理介面讓政府端無需技術背景即可操作。 |
| **市場可行性** | 算力套利人力（雲端推論成本 << 真人諮詢工時）；活化政策預算（自動媒合提升觸及率）。 |
| **擴展性** | 插件式 Agent 架構：新增資源 = 新增 Agent。可從台北市擴展到全台，未來可接入民間企業。 |
| **社會影響力** | 當一個青年因搜尋成本太高而放棄，不僅是個人損失，更是整個社會的損失。MATCHA 將搜尋成本降至接近零，讓每個人都能被看見。 |

---

## 8. 結論

政府擁有豐富的資源，青年擁有迫切的需求，但中間缺少的從來不是「搜尋引擎」，而是「社群」。

**MATCHA 的核心理念是：讓 AI Agent 建立一個 24 小時運作的職涯社群，代替青年去社交、代替政府去尋找。在這個社群中，每個人都能被看見，每個資源都能找到它該去的地方。**
